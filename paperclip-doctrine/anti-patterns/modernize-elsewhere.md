# Modernize Elsewhere, Migrate Later

Building replacement systems in isolation before improving current systems. This approach often wastes resources and delays addressing real problems.

## Why It's Problematic

Parallel development splits attention and resources while delaying necessary improvements to existing systems. Migration rarely succeeds as planned, leaving organizations with two problems instead of one.

## Signs of This Anti-Pattern

- Greenfield projects that duplicate existing functionality
- Parallel teams working on "next generation" solutions
- Current system maintenance deprioritized for future migration
- Assumptions about seamless migration between systems
- Investment in temporary solutions with planned obsolescence

## How to Avoid

- Focus improvement efforts on current systems first
- Evaluate every new component for integration with existing infrastructure
- Plan evolutionary enhancements rather than revolutionary replacements
- Validate migration assumptions with small experiments
- Maintain investment in current systems throughout any parallel development

## Recovery Strategies

- Halt parallel development to focus on current system improvements
- Redirect resources toward enhancing existing capabilities
- Create integration paths between new and old systems
- Establish clear criteria for when migration makes sense
- Plan for coexistence rather than replacement whenever possible
