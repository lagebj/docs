## Overview

The Grounded Systems Design System provides a comprehensive set of guidelines, components, and patterns to ensure consistent, accessible, and beautiful user experiences across all documentation pages.

## Foundations

### Color Palette

Our color system is designed to convey trust, clarity, and professionalism while maintaining accessibility standards.

| Token | Value | Usage |
|-------|-------|-------|
| `gs-brand-blue` | `#1e40af` | Primary actions, links, headings |
| `gs-brand-slate` | `#334155` | Body text, secondary UI |
| `gs-border-muted` | `rgba(51, 65, 85, 0.2)` | Borders, dividers |
| `gs-surface-soft` | `linear-gradient(180deg, rgba(219, 234, 254, 0.5) 0%, rgba(248, 250, 252, 0.96) 100%)` | Card backgrounds |
| `gs-surface-band` | `linear-gradient(180deg, rgba(191, 219, 254, 0.42) 0%, rgba(241, 245, 249, 0.92) 100%)` | Section bands |

### Typography

We use a system UI font stack with `Inter` as a fallback for optimal readability.

```css
font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, Oxygen, Ubuntu, Cantarell, 'Open Sans', 'Helvetica Neue', sans-serif;
```

**Heading Hierarchy:**

| Level | Size | Weight | Mobile Size |
|-------|------|--------|-------------|
| H1 | 2.5rem | 700 | 2rem |
| H2 | 1.8rem | 700 | 1.6rem |
| H3 | 1.4rem | 600 | 1.3rem |
| H4 | 1.15rem | 600 | 1.1rem |

### Spacing System

We use a consistent spacing scale based on `0.5rem` increments:

| Token | Value | Class |
|-------|-------|-------|
| `space-xs` | 0.25rem | N/A |
| `space-sm` | 0.5rem | N/A |
| `space-md` | 1rem | N/A |
| `space-lg` | 1.5rem | N/A |
| `space-xl` | 2rem | N/A |
| `space-vertical` | `clamp(2.5rem, 2rem + 1vw, 3.5rem)` | `spacing-vertical` |

1. Minimum contrast ratio of 4.5:1 for text
2. Focus indicators for interactive elements
3. Semantic HTML structure
4. Proper ARIA attributes where needed
5. Keyboard navigation support

## Accessibility

All components comply with WCAG AA standards to ensure accessibility for all users:

1. **Color Contrast** - Minimum contrast ratio of 4.5:1 for text
2. **Focus Indicators** - Visible focus states for all interactive elements
3. **Semantic HTML** - Proper use of heading levels and semantic elements
4. **ARIA Attributes** - Appropriate use of ARIA roles and properties where needed
5. **Keyboard Navigation** - Full keyboard operability for all interactive components

### Focus States

```css
.gs-focus-outline {
  outline: 2px solid var(--gs-brand-blue);
  outline-offset: 2px;

```

### Screen Reader Support

For components that need additional screen reader support:

```html
<!-- Example: Card with aria-labelledby -->
<div class="gs-card" role="region" aria-labelledby="card-heading-1">
  <h3 id="card-heading-1">Card Title</h3>
  <p>Card content goes here with supporting details.</p>
</div>

<!-- Example: Button with aria-describedby -->
<button class="gs-button gs-button-primary" aria-describedby="save-button-desc">
  Save Changes
</button>
<div id="save-button-desc" class="sr-only">
  Saves your changes to the server
</div>
```

### Reduced Motion

Support for users who prefer reduced motion:

@media (prefers-reduced-motion: reduce) {
  .gs-button,
  .gs-card {
    transition: none;
  }
  
  .gs-card:hover {
    transform: none;
  }

## Responsive Design

All components are mobile-first and responsive, with breakpoints optimized for various device sizes:

### Breakpoints

| Device | Breakpoint |
|--------|------------|
| Small (mobile) | `max-width: 480px` |
| Medium (tablet) | `min-width: 481px` to `max-width: 768px` |
| Large (desktop) | `min-width: 769px` to `max-width: 1024px` |
| Extra Large (wide) | `min-width: 1025px` |

### Media Queries

```css
/* Mobile first - base styles */

/* Small devices (landscape phones, 480px and up) */
@media (min-width: 480px) {
  .gs-container {
    max-width: 480px;
  }
}

/* Medium devices (tablets, 768px and up) */
@media (min-width: 768px) {
  .gs-container {
    max-width: 768px;
  }
  
  /* Enhancements for tablet */
  .gs-card {
    padding: 2rem;
  }
}

/* Large devices (desktops, 1024px and up) */
@media (min-width: 1024px) {
  .gs-container {
    max-width: 1024px;
  }
  
  /* Desktop enhancements */
  .gs-card {
    padding: 2.5rem;
  }
}

/* Extra large devices */
@media (min-width: 1200px) {
  .gs-container {
    max-width: 1200px;
  }
}
```

## Implementation Guidelines

### CSS Variables

All design tokens are available as CSS variables:

```css
:root {
  --gs-brand-blue: #1e40af;
  --gs-brand-slate: #334155;
  --gs-border-muted: rgba(51, 65, 85, 0.2);
  --gs-surface-soft: linear-gradient(180deg, rgba(219, 234, 254, 0.5) 0%, rgba(248, 250, 252, 0.96) 100%);
  --gs-space-vertical: clamp(2.5rem, 2rem + 1vw, 3.5rem);
}
```

### Usage in Components

1. Always use CSS variables instead of hardcoded values
2. Maintain consistent spacing using the spacing system
3. Ensure adequate touch targets (minimum 44px)
4. Test components in all supported viewports
5. Validate color contrast ratios

## Component Usage Guidelines

### When to Use Each Component

#### Buttons
- **Primary**: Main actions, form submissions, primary navigation
- **Secondary**: Alternative actions, less important actions
- **Danger**: Destructive actions like delete or remove

#### Cards
- Grouping related content
- Feature highlights
- Content previews

#### Alerts
- **Info**: General information, tips, neutral messages
- **Success**: Confirmation messages, successful actions
- **Warning**: Cautionary messages, validation warnings
- **Danger**: Error messages, destructive action confirmations

#### Tags
- Categorizing content
- Status indicators
- Filtering mechanisms

### Best Practices

1. **Consistency**: Use the same components for similar purposes throughout the site
2. **Hierarchy**: Establish clear visual hierarchy with spacing and sizing
3. **Progressive Enhancement**: Ensure components work without JavaScript
4. **Performance**: Minimize CSS bloat by using only necessary components
5. **Maintenance**: Document customizations and overrides

### Customization

To customize components for specific use cases:

1. Use modifier classes for variations (e.g., `.gs-button-lg`)
2. Extend existing components rather than creating new ones
3. Maintain the core design language when extending

### Contributing New Components

When adding new components to the design system:

1. Ensure they follow the existing design language
2. Provide comprehensive documentation
3. Include accessibility considerations
4. Test across all supported browsers and devices
5. Add to this documentation with usage guidelines
